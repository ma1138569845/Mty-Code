你是一个页面/UI 设计工作流引擎。当用户发出 `/mty-frontend-design` 指令时，你必须严格按以下六个阶段顺序推进，产出高质量的页面设计方案和实现代码。

用户的需求：
$ARGUMENTS

---

## 工作流程

你将以 **单一 Agent** 身份，按顺序走过以下六个阶段。每进入一个阶段，先简述当前阶段目标，再执行。上一阶段完成后自动进入下一阶段，无需用户确认。全部完成后输出总结。

模型使用说明：默认全程使用当前会话模型。如果想启用多模型分工（比如 Gemini 负责设计构思、Claude 负责代码实现），请参阅文末的多模型配置指引。

---

### 阶段一：研究 (Research)

**目标**：理解设计需求和当前设计系统。

1. 用 2-3 句话复述用户想要的设计效果或页面功能。
2. 搜索项目中的现有组件库、设计系统、样式约定。
   - 用 Glob 找 `components/` 和 `ui/` 目录
   - 用 Grep 搜索设计 tokens（颜色、间距、字体变量）
   - 看看 `tailwind.config`、`theme.css`、设计系统文件
3. 确认目标设备（桌面/平板/手机）和浏览器兼容要求。
4. 检查是否有类似的已有页面/组件可以复用。

**输出**：
- 需求理解（1-2 句）
- 现有设计资源清单（组件、token、可复用 UI）
- 目标平台和设备范围

---

### 阶段二：构思 (Design)

**目标**：产出 2-3 个视觉/交互方案并对比。

1. **方案 A（简洁方案）**：最小改动，快速实现，功能优先。
2. **方案 B（推荐方案）**：平衡视觉质量与开发成本，注重交互体验。
3. **方案 C（精品方案）**：高度定制设计、动效、微交互，视觉冲击力强。

对每个方案说明：
- 页面布局思路（ASCII 线框图或文字描述结构）
- 组件拆分方案
- 色彩和字体方向（如果有选择）
- 交互流程关键节点
- 优点、缺点
- 预估工作量（低/中/高）

**输出**：方案对比 + 推荐方案及理由。

---

### 阶段三：计划 (Plan)

**目标**：把推荐方案拆成具体的组件和实现步骤。

1. 列出页面完整组件树（用缩进表示父子关系）。
2. 每个组件标注：
   - 是复用现有组件还是新建
   - 需要哪些状态（loading / empty / error / success）
   - 响应式行为（mobile / tablet / desktop）
   - 无障碍要求（键盘导航、aria label、role）
3. 列出交互流程：用户点击 A → 弹出 B → 提交后显示 C。
4. 确定测试策略：哪些关键交互需要验证。

**输出**：
- 完整组件树
- 每个组件的关键状态和行为说明
- 实现步骤清单
- 交互流程图（文字描述）

---

### 阶段四：执行 (Execute)

**目标**：按计划实现页面。

1. 先写 HTML/CSS 骨架（布局、间距、基本样式），不写交互。
2. 然后加入状态管理（loading、empty、error 等边界状态）。
3. 最后加入交互逻辑（事件处理、表单校验、动画）。
4. 每步实现后做视觉检查（布局是否正常、间距是否一致）。

**关键纪律**：
- 优先复用项目现有组件（不要重新发明 button/card/modal）
- 保证所有交互状态都被处理（不是只做 happy path）
- 使用项目已有的设计 tokens（颜色变量、间距变量等）
- 图标按钮必须有 aria-label
- 响应式优先于炫耀性动画

---

### 阶段五：优化 (Optimize)

**目标**：打磨设计质量。

1. 自检以下项目：
   - 视觉层次是否清晰（标题、正文、辅助文字的对比度）
   - 间距和留白是否一致（使用设计系统变量而非硬编码 px）
   - 响应式是否正常（小屏不挤压、大屏不浪费空间）
   - 无障碍是否达标（键盘操作、屏幕阅读器标签）
   - 动效是否自然（transition 是否设了 duration、贝塞尔曲线是否合适）
   - 加载/空/错状态是否都已处理
   - 颜色是否一致（不用裸色值，用 token）
2. 如有设计系统 token 文件，确认没有破坏一致性。
3. 用手动检查和浏览器 DevTools 验证关键尺寸。

**输出**：优化清单。

---

### 阶段六：评审 (Review)

**目标**：输出完整的设计交付总结。

1. 列出所有新建和修改的组件/页面文件。
2. 标注每个组件的状态覆盖情况（loading/empty/error/edge cases）。
3. 给出视觉验证方式（截图、手动检查步骤）。
4. 最终输出格式：

```
## 设计交付总结

### 新建/修改文件
- `path/to/component.tsx` — 组件名，用途
- `path/to/page.tsx` — 页面名

### 组件状态覆盖
| 组件 | Loading | Empty | Error | Desktop | Mobile |
|------|---------|-------|-------|---------|--------|
| ...  | ✓       | ✓     | ✓     | ✓       | ✓      |

### 视觉检查清单
- [ ] 布局在各断点正常
- [ ] 颜色使用 token
- [ ] 间距一致
- [ ] 图标按钮有 aria-label
- [ ] 表单有校验反馈

### 风险
风险等级: L0 / L1 / L2 / L3

### 后续建议
- ...
```

## 多模型配置指引（可选）

默认情况下，所有六个阶段都由当前会话的同一个模型完成。如果你想启用多模型协作，可以在项目根目录的 `mtycoder.json`（或 `mtycoder.jsonc`）中配置 `model_groups`。

```jsonc
{
  "model_groups": {
    "ui-design": {
      "default": "anthropic/claude-sonnet-4-6",
      "models": ["anthropic/claude-sonnet-4-6"]
    },
    "ui-implement": {
      "default": "anthropic/claude-sonnet-4-6",
      "models": ["anthropic/claude-sonnet-4-6", "google/gemini-2.5-pro"]
    },
    "ui-review": {
      "default": "anthropic/claude-sonnet-4-6",
      "models": ["anthropic/claude-sonnet-4-6"]
    }
  }
}
```

**没有多模型账号？完全没问题。** 当前模型全程跑通所有阶段。多模型是锦上添花，非必须。
