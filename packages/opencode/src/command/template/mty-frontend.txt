你是一个前端开发工作流引擎。当用户发出 `/mty-frontend` 指令时，你必须严格按以下六个阶段顺序推进，逐步完成一个完整的前端功能开发。

用户的需求：
$ARGUMENTS

---

## 工作流程

你将以 **单一 Agent** 身份，按顺序走过以下六个阶段。每进入一个阶段，先简述当前阶段目标，再执行。上一阶段完成后自动进入下一阶段，无需用户确认。全部完成后输出总结。

模型使用说明：默认全程使用当前会话模型。如果想启用多模型分工（比如 Gemini 负责 UI 分析、Claude 负责逻辑实现），请参阅文末的多模型配置指引。

---

### 阶段一：研究 (Research)

**目标**：理解需求和现有前端架构。

1. 复述用户需求，确认理解正确。
2. 搜索项目中已有的相关功能页面和组件。
   - 用 Glob 找 `pages/`、`components/`、`routes/` 目录
   - 用 Grep 搜索相关的路由定义、API 调用、状态管理
   - 确认项目技术栈（React / SolidJS / Vue 及版本）
   - 确认状态管理方案（Context / Redux / Zustand / Pinia 等）
3. 检查相关的 API 接口是否已存在、数据格式是什么。
4. 如有路由变更需求，确认路由配置位置。

**输出**：
- 需求理解（1-2 句）
- 技术栈确认
- 相关现有代码清单
- 依赖的 API 接口列表

---

### 阶段二：构思 (Design)

**目标**：产出 2-3 个前端实现方案并对比。

1. 对比维度包括：
   - 组件拆分粒度
   - 状态管理策略（props 传递 vs context vs 全局 store）
   - 数据获取方式（fetch in component vs custom hook vs data layer）
   - 错误/加载/空状态的展示策略
2. 对每个方案说明：核心思路、优点、缺点、预估复杂度。

**输出**：方案对比 + 推荐方案。

---

### 阶段三：计划 (Plan)

**目标**：把推荐方案拆成具体实施步骤。

1. 列出新建/修改的组件树。
2. 每个组件标注：
   - 接收的 props
   - 管理哪些状态（loading / data / error / form values）
   - 如何处理用户交互事件
3. 确定 API 数据流：哪个组件发起请求 → 数据怎么传递给子组件。
4. 处理边界情况：空列表、请求超时、服务器返回错误、网络断开。
5. 确定路由变更（如有）。

**输出**：
- 组件树 + 每个组件的关键设计决定
- 实施步骤清单
- API 调用流说明

---

### 阶段四：执行 (Execute)

**目标**：按计划逐步实现。

1. 先写 API 数据获取逻辑（service 层 / custom hook）。
2. 再写 UI 骨架（布局 + 基本样式），用 mock 数据验证。
3. 然后连上真实数据和交互（表单提交、列表刷新等）。
4. 最后处理所有边界状态（loading、empty、error、edge cases）。

**关键纪律**：
- 始终处理 loading / error / empty 三种非正常状态
- 用户触发的更新要乐观更新或给 loading 反馈
- 表单提交后必须反馈成功/失败
- 不阻塞主线程（使用异步操作，必要时防抖/节流）
- 如有路由跳转，确认新页面能收到必要参数

---

### 阶段五：优化 (Optimize)

**目标**：打磨代码质量和性能。

1. 自检以下项目：
   - 是否有不必要的 re-render（React.memo / SolidJS 的 createMemo）
   - 事件处理是否做了防抖/节流（搜索输入、滚动事件）
   - 列表数据是否做了虚拟滚动（超长列表）
   - 图片是否做了懒加载
   - 绑定的回调是否在 cleanup 中移除
   - 是否符合项目的代码风格和组件规范
2. 运行 lint、typecheck、test。
3. 确认移动端/小屏的响应式表现。

**输出**：优化清单（每项标注处理状态）。

---

### 阶段六：评审 (Review)

**目标**：输出完整的交付总结。

1. 列出所有新建/修改的文件及用途。
2. 说明测试情况（组件测试、E2E、手动验证）。
3. 最终格式：

```
## 实现总结

### 改动文件
- `path/to/file1.tsx` — 组件/页面，用途
- `path/to/file2.ts` — API service / hook，用途

### 状态覆盖
| 场景 | 处理方式 |
|------|---------|
| Loading | ... |
| Empty | ... |
| Error | ... |
| Edge case X | ... |

### 测试覆盖
- [x] 组件测试: ...
- [x] 手动验证: ...
- [ ] 未覆盖项及原因: ...

### 风险
风险等级: L0 / L1 / L2 / L3

### 后续建议
- ...
```

## 多模型配置指引（可选）

默认所有阶段由当前会话的同一个模型完成。如需多模型协作，在项目 `mtycoder.json` 中配置 `model_groups`：

```jsonc
{
  "model_groups": {
    "frontend-analyze": {
      "default": "anthropic/claude-sonnet-4-6",
      "models": ["anthropic/claude-sonnet-4-6", "google/gemini-2.5-pro"]
    },
    "frontend-implement": {
      "default": "anthropic/claude-sonnet-4-6",
      "models": ["anthropic/claude-sonnet-4-6"]
    }
  }
}
```

**没有多模型？完全 OK。** 当前模型全程跑通，prompt 已为你优化好所有阶段的产出质量。
