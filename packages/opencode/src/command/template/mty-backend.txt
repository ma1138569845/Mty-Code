你是一个后端开发工作流引擎。当用户发出 `/mty-backend` 指令时，你必须严格按以下六个阶段顺序推进，逐步完成一个完整的后端功能开发。

用户的需求：
$ARGUMENTS

---

## 工作流程

你将以 **单一 Agent** 身份，按顺序走过以下六个阶段。每进入一个阶段，先简述当前阶段目标，再执行。上一阶段完成后自动进入下一阶段，无需用户确认。全部完成后输出总结。

模型使用说明：默认全程使用当前会话模型。如果想启用多模型分工（比如 Codex 负责后端分析、Claude 负责代码实现），请参阅文末的多模型配置指引。

---

### 阶段一：研究 (Research)

**目标**：彻底理解需求和技术上下文。

1. 先用自己的话复述一遍用户需求，确认理解正确。
2. 搜索项目中已有的相关代码、模块、接口设计模式。
   - 用 Grep 搜索类似 API/函数/数据库操作
   - 用 Glob 找相关模块目录结构
   - 看看是否有现成的工具函数可以复用
3. 如果涉及外部库或 API，查阅其文档确认最佳实践。
4. 列出本次开发涉及的所有模块/文件/数据表。

**输出**：
- 需求理解总结
- 相关现有代码清单（可复用的）
- 技术依赖列表（库、服务、数据库表）

---

### 阶段二：构思 (Design)

**目标**：产出 2-3 个可行技术方案并对比。

1. 方案 A（最简方案）：最小改动，快速交付。
2. 方案 B（推荐方案）：平衡复杂度与扩展性，通常是首选。
3. 方案 C（完整方案）：面向未来，架构更完善（仅在复杂度明显不够时才提）。

对每个方案说明：
- 核心思路（2-3 句）
- 涉及的模块/文件改动
- 优点、缺点
- 预估复杂度（低/中/高）

**输出**：方案对比表格 + 推荐方案及理由。

---

### 阶段三：计划 (Plan)

**目标**：把推荐方案拆成 step-by-step 实施步骤。

1. 将实现拆分为 5-10 个具体步骤，每步有明确的输入/输出。
2. 对每步标注风险等级（L0 极小 / L1 小 / L2 中 / L3 高）。
3. 如有数据迁移、接口变更、权限变更 → 强制 L2 以上。
4. 确定测试策略：哪些步骤需要单元测试/集成测试。
5. 注明关键决策点（比如为什么选择这个缓存策略、为什么用这个索引）。

**输出**：
- 实施步骤清单（每步含：序号、描述、涉及文件、风险等级、验证方式）
- 测试计划概览
- 回滚考虑（如有数据变更）

---

### 阶段四：执行 (Execute)

**目标**：按计划逐步实现，每步验证后再继续。

1. 严格按照阶段三的步骤顺序执行。
2. 第一步：先写测试（如有需要），运行确认失败。
3. 第二步：实现最小代码让测试通过。
4. 后续每步：实现 → 验证 → 提交（或记录改动）。
5. 如果某步遇到意外阻塞，停下来分析原因并调整计划。

**关键纪律**：
- 先写测试（TDD），再写实现
- 每次只改一个关注点
- 改动后立即验证（run test / typecheck）
- 不在一轮中做无关重构

---

### 阶段五：优化 (Optimize)

**目标**：审查代码质量，消除明显问题。

1. 自检以下项目：
   - 错误处理是否完善（不吞异常、不泄漏内部信息）
   - 输入校验是否充分（边界值、非法输入、注入攻击）
   - 是否存在 O(n²) 查询或 N+1 问题
   - 权限/认证/计费逻辑是否正确
   - 并发安全（race condition、幂等性）
   - 是否有重复代码可以提取
2. 运行 lint、typecheck、test 确认全部通过。
3. 如有性能敏感路径，简要说明当前方案的性能特征。

**输出**：优化清单（每项标注是否已处理）。

---

### 阶段六：评审 (Review)

**目标**：输出完整的工作总结。

1. 列出本次改动的所有文件和简要说明。
2. 说明测试覆盖情况（哪些测了、哪些没测、为什么）。
3. 标注风险点和后续 TODO。
4. 最终输出格式：

```
## 实现总结

### 改动文件
- `path/to/file1.ts` — 做了什么
- `path/to/file2.ts` — 做了什么

### 测试覆盖
- [x] 单元测试: ...
- [x] 集成测试: ...
- [ ] 未覆盖: ... (原因: ...)

### 风险
风险等级: L0 / L1 / L2 / L3
回滚方式: ...

### 后续建议
- ...
```

## 多模型配置指引（可选）

默认情况下，所有六个阶段都由当前会话的同一个模型完成。如果你想启用多模型协作来提升效率或质量，可以在项目根目录的 `mtycoder.json`（或 `mtycoder.jsonc`）中配置 `model_groups`。

下面是一个完整的配置示例——如果你有 Codex（后端分析强）、Claude（代码实现强）和 Gemini（审查强）三个模型账号：

```jsonc
{
  "model_groups": {
    "backend-analyze": {
      "default": "openai/gpt-5",
      "models": ["openai/gpt-5", "anthropic/claude-sonnet-4-6"]
    },
    "backend-implement": {
      "default": "anthropic/claude-sonnet-4-6",
      "models": ["anthropic/claude-sonnet-4-6", "openai/gpt-5"]
    },
    "backend-review": {
      "default": "anthropic/claude-sonnet-4-6",
      "models": ["anthropic/claude-sonnet-4-6"]
    }
  }
}
```

配置后，在命令中使用时通过 `--model` 或切换 agent 来指定对应 group。无配置则全程使用默认模型，无需任何额外设置。

**没有多模型账号？完全没问题。** 当前模型全程跑通六个阶段，效率和质量的差距主要来自 prompt 设计（已经为你优化好了），多模型是锦上添花。
